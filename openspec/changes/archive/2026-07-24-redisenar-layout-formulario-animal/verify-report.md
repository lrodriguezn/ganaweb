## Findings

### CRITICAL

**1. CA-UI-014 spec violation — missing `*` on Sexo, Origen, Fecha de nacimiento**

The spec (lines 35-37, 60-62, 474-475) is unambiguous:
> Only **Código**, **Sexo**, **Origen**, and **Fecha de nacimiento** MUST carry the visible asterisk; every other field MUST NOT show `*` or a "obligatorio" marker next to its label (CA-UI-014).
> THEN the asterisk `*` is visible only on Código, Sexo, Origen, and Fecha de nacimiento

The implementation only has the asterisk on `Código` (baked into the label string at `animal-crud.tsx:634` as `"Código *"`). The other three required fields **do not** display the `*`:

- **Sexo** (`animal-crud.tsx:637`, label `"Sexo"`): `renderSexoField` (line 1131) uses `<Label htmlFor=...>{field.label}</Label>` on mobile (line 1147) and `CatalogSelectField` on desktop without `required`, so no `*` is rendered.
- **Origen** (`animal-crud.tsx:642`, label `"Origen"`): `renderOrigenField` (line 1199) delegates to `OrigenField` (line 1714) which renders `<Label htmlFor=...>{label}</Label>` (line 1730) without any asterisk.
- **Fecha de nacimiento** (`animal-crud.tsx:639`, label `"Fecha de nacimiento"`): `FechaNacimientoField` (line 1761) renders `<Label htmlFor=...>{label}</Label>` (line 1781) without any asterisk.

The fallback `Field` component (line 1561) and the `OrigenField` / `FechaNacimientoField` wrappers all skip rendering the `*` even when the underlying input is `required`. Only `CatalogSelectField` (line 1673) renders `<span aria-hidden="true"> *</span>` from the `required` prop.

**Existing test coverage is insufficient**: the only CA-UI-014 test (`animal-ui.test.tsx:1560`) verifies the **negative** (Tipo de explotación has NO asterisk). The existing label-presence test (line 175-193) uses `getByLabelText("Sexo")` / `getByLabelText("Origen")` / `getByLabelText("Fecha de nacimiento")` — without the trailing `*` — which would **fail** if the spec-compliant `*` were added. The test was written to match the buggy behavior.

**Why this matters**: this is a normative rule (CA-UI-014, the corrected requirement). The user-visible label is the contract — missing asterisks tell the user a field is optional when it is actually required. §3.1 of `crud_animales.md` v1.5 backs this.

### WARNING

**1. Unused destructured `fields` (animal-crud.tsx:789)** — `fields` is destructured from `useAnimalForm(...)` but never read. The refactor replaced `fields.map(renderAnimalFormField)` with section-aware `renderFieldByName`, so `fields` is dead code. `biome lint` blocks the merge. Trivial fix: remove from the destructure.

**2. `useEffect` missing `collapsibleOpen` and `detailFieldErrors` in dependency array (animal-crud.tsx:868-880)** — `biome lint` warns. The effect intentionally re-runs only when `JSON.stringify(detailFieldErrors)` changes (using a JSON-stringify key) and reads `collapsibleOpen` via closure. The current code is **correct at runtime** (stringify key already forces re-evaluation when errors change) but the dependency list is misleading. The cleanest fix: drop `collapsibleOpen` from the body (always set `true`) since the force-open is idempotent, or use a `useRef` for the boolean check.

**3. CA-UI-010 focus/scroll assertion missing (T-303)** — Task T-303 explicitly required:
> when `fieldErrors` intersects `DETAIL_FIELD_NAMES`, Collapsible auto-opens and `document.getElementsByName(name)[0].scrollIntoView` is called with NO `behavior: "smooth"` (assert call args)

The implementation does call `scrollIntoView({ block: "nearest" })` at line 876 (correct, no `behavior: "smooth"`), but the test at `animal-ui.test.tsx:1434` only asserts that the collapsible is open and the error text is visible. The `scrollIntoView` call args are not stubbed/asserted. If a future refactor drops the focus call, no test will catch it.
